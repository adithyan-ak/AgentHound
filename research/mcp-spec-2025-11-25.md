## 2. Pagination Protocol

Cursor-based (not page-numbered). Page size is server-determined; clients MUST NOT assume a fixed size.

**Operations supporting pagination:**
- `resources/list`
- `resources/templates/list`
- `prompts/list`
- `tools/list`
- `tasks/list`

**Response format:** Result object includes an optional `nextCursor` string field. Absence of `nextCursor` means end of results.

```json
{
  "jsonrpc": "2.0",
  "id": "123",
  "result": {
    "resources": [...],
    "nextCursor": "eyJwYWdlIjogM30="
  }
}
```

**Request with cursor:**
```json
{
  "jsonrpc": "2.0",
  "id": "124",
  "method": "resources/list",
  "params": {
    "cursor": "eyJwYWdlIjogMn0="
  }
}
```

**Cursor type:** `string` (opaque). Defined in schema as `export type Cursor = string;`

**Rules:**
- Clients MUST treat cursors as opaque tokens
- Clients MUST NOT parse, modify, or persist cursors across sessions
- Invalid cursors SHOULD result in error code `-32602` (Invalid params)
- Servers SHOULD provide stable cursors

## 3. Streamable HTTP Transport

Replaces the HTTP+SSE transport from protocol version 2024-11-05.

### Server Endpoint

Server provides a single HTTP endpoint (e.g., `https://example.com/mcp`) that supports both POST and GET methods.

### Sending Messages to the Server (Client -> Server via POST)

- Client sends every JSON-RPC message as a new **HTTP POST** to the MCP endpoint.
- Request body: a single JSON-RPC request, notification, or response.
- **Required headers:**
  - `Accept: application/json, text/event-stream`
  - `Content-Type: application/json`
  - `MCP-Protocol-Version: 2025-11-25` (on all requests after initialization)
  - `MCP-Session-Id: <session-id>` (if server assigned one during initialization)
  - `Authorization: Bearer <token>` (if using OAuth)

**If input is a notification or response:** Server returns `202 Accepted` with no body.

**If input is a request:** Server returns either:
- `Content-Type: application/json` -- single JSON-RPC response object, OR
- `Content-Type: text/event-stream` -- SSE stream containing:
  1. An initial SSE event with an event ID and empty `data` field (to prime reconnection)
  2. Zero or more JSON-RPC requests/notifications from the server
  3. The JSON-RPC response to the original request
  4. Stream termination after the response

### Listening for Server Messages (Client -> Server via GET)

- Client MAY issue HTTP GET to the MCP endpoint to open an SSE stream for server-initiated messages.
- **Required headers:**
  - `Accept: text/event-stream`
  - `MCP-Protocol-Version: 2025-11-25`
  - `MCP-Session-Id: <session-id>` (if assigned)
- Server returns `Content-Type: text/event-stream` or `405 Method Not Allowed`.
- Server MAY send JSON-RPC requests and notifications on this stream.
- Server MUST NOT send JSON-RPC responses on this stream (unless resuming a previous stream).

### Session Management

- Server MAY assign a session ID via `MCP-Session-Id` header on the HTTP response containing the `InitializeResult`.
- Session ID: visible ASCII characters (0x21-0x7E), globally unique, cryptographically secure.
- Client MUST include `MCP-Session-Id` on all subsequent HTTP requests.
- Server returns `400 Bad Request` for missing session ID, `404 Not Found` for expired/terminated sessions.
- Client sends `HTTP DELETE` to MCP endpoint with `MCP-Session-Id` to terminate a session.

### Resumability

- Servers MAY attach `id` fields to SSE events.
- On disconnection, client reconnects via HTTP GET with `Last-Event-ID` header.
- Server MAY replay missed messages for that specific stream.
- Server MUST NOT replay messages from a different stream.

### Security

- Servers MUST validate `Origin` header; respond `403 Forbidden` if invalid.
- Locally-running servers SHOULD bind only to `127.0.0.1`.

## 4. stdio Transport

- Client launches MCP server as a subprocess.
- Server reads JSON-RPC from `stdin`, writes JSON-RPC to `stdout`.
- **Message framing: newline-delimited JSON-RPC.** Messages MUST NOT contain embedded newlines.
- Server MAY write UTF-8 strings to `stderr` for logging.
- Server MUST NOT write non-MCP content to `stdout`.
- Client MUST NOT write non-MCP content to server's `stdin`.

**Shutdown sequence:**
1. Client closes input stream to server
2. Wait for server to exit
3. Send `SIGTERM` if server doesn't exit in reasonable time
4. Send `SIGKILL` if still alive after `SIGTERM`

## 5. `notifications/initialized`

Sent by client after successful `initialize` response. No params, no `id` (it's a notification).

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/initialized"
}
```

**Sequencing rules:**
- Client SHOULD NOT send requests other than `ping` before server responds to `initialize`.
- Server SHOULD NOT send requests other than `ping` and `logging` before receiving `notifications/initialized`.

## 7. Tool Annotations

From `ToolAnnotations` interface in schema.ts. These are **hints** -- not guaranteed to be truthful. Clients MUST consider them untrusted unless from trusted servers.

```typescript
export interface ToolAnnotations {
  title?: string;
  readOnlyHint?: boolean;     // default: false
  destructiveHint?: boolean;  // default: true
  idempotentHint?: boolean;   // default: false
  openWorldHint?: boolean;    // default: true
}
```

| Annotation         | Default | Semantics                                                             |
|--------------------|---------|-----------------------------------------------------------------------|
| `readOnlyHint`     | `false` | If true, tool does NOT modify its environment                          |
| `destructiveHint`  | `true`  | If true, tool MAY perform destructive updates. Only meaningful when `readOnlyHint == false` |
| `idempotentHint`   | `false` | If true, repeated calls with same args have no additional effect. Only meaningful when `readOnlyHint == false` |
| `openWorldHint`    | `true`  | If true, tool may interact with an open world of external entities (e.g., web search). If false, domain is closed (e.g., memory tool) |

**Example in tool definition:**
```json
{
  "name": "delete_file",
  "description": "Deletes a file",
  "inputSchema": { "type": "object", "properties": { "path": { "type": "string" } }, "required": ["path"] },
  "annotations": {
    "readOnlyHint": false,
    "destructiveHint": true,
    "idempotentHint": true,
    "openWorldHint": false
  }
}
```

**Tool execution field (new in 2025-11-25):**
```json
{
  "execution": {
    "taskSupport": "optional"
  }
}
```
Values: `"forbidden"` (default), `"optional"`, `"required"`

## 8. OAuth 2.1 Authorization Flow

**Applies to:** HTTP-based transports only. stdio transports SHOULD retrieve credentials from the environment.

**Standards:**
- OAuth 2.1 (draft-ietf-oauth-v2-1-13)
- RFC8414 (OAuth 2.0 Authorization Server Metadata)
- RFC7591 (Dynamic Client Registration)
- RFC9728 (OAuth 2.0 Protected Resource Metadata)
- OAuth Client ID Metadata Documents (draft-ietf-oauth-client-id-metadata-document-00)

### Discovery Flow

1. Client sends request to MCP server without auth.
2. Server returns `401 Unauthorized` with:
   ```
   WWW-Authenticate: Bearer resource_metadata="https://mcp.example.com/.well-known/oauth-protected-resource",
                            scope="files:read"
   ```
3. Client fetches Protected Resource Metadata (RFC9728) from the `resource_metadata` URL or well-known URI:
   - `https://example.com/.well-known/oauth-protected-resource/public/mcp` (path-based)
   - `https://example.com/.well-known/oauth-protected-resource` (root fallback)
4. Protected Resource Metadata MUST include `authorization_servers` field.
5. Client discovers authorization server metadata via (in priority order):
   - With path: `https://auth.example.com/.well-known/oauth-authorization-server/tenant1`
   - With path: `https://auth.example.com/.well-known/openid-configuration/tenant1`
   - With path: `https://auth.example.com/tenant1/.well-known/openid-configuration`
   - Without path: `https://auth.example.com/.well-known/oauth-authorization-server`
   - Without path: `https://auth.example.com/.well-known/openid-configuration`

### Client Registration Priority

1. Pre-registered client credentials (if available)
2. Client ID Metadata Documents (if auth server advertises `client_id_metadata_document_supported: true`)
3. Dynamic Client Registration (RFC7591 fallback, if `registration_endpoint` present)
4. Prompt user to enter client information

### Client ID Metadata Document Example

```json
{
  "client_id": "https://app.example.com/oauth/client-metadata.json",
  "client_name": "Example MCP Client",
  "client_uri": "https://app.example.com",
  "logo_uri": "https://app.example.com/logo.png",
  "redirect_uris": [
    "http://127.0.0.1:3000/callback",
    "http://localhost:3000/callback"
  ],
  "grant_types": ["authorization_code"],
  "response_types": ["code"],
  "token_endpoint_auth_method": "none"
}
```

### Token Usage

- Client MUST use `Authorization: Bearer <token>` header on EVERY HTTP request.
- Client MUST include `resource` parameter (RFC 8707) in auth and token requests.
- Tokens MUST NOT be in URI query strings.
- Client MUST use PKCE with `S256` challenge method.

### Incremental Scope (Step-Up Authorization)

When server returns:
```
HTTP/1.1 403 Forbidden
WWW-Authenticate: Bearer error="insufficient_scope",
                         scope="files:read files:write user:profile",
                         resource_metadata="https://mcp.example.com/.well-known/oauth-protected-resource",
                         error_description="Additional file write permission required"
```

Client SHOULD re-authorize with the expanded scope set and retry.

### Scope Selection Strategy (initial)

1. Use `scope` from `WWW-Authenticate` header in the initial 401 response
2. Fallback: use all scopes from `scopes_supported` in Protected Resource Metadata
3. If `scopes_supported` is undefined, omit `scope` parameter

## 9. Tasks (Experimental, new in 2025-11-25)

Durable state machines for long-running/deferred operations. Either client or server can be requestor/receiver.

### Task Lifecycle

```
working --> input_required --> working  (loop)
working --> completed (terminal)
working --> failed (terminal)
working --> cancelled (terminal)
input_required --> completed (terminal)
input_required --> failed (terminal)
input_required --> cancelled (terminal)
```

### Creating a Task

Client sends a request with `task` field in params:
```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "get_weather",
    "arguments": { "city": "New York" },
    "task": {
      "ttl": 60000
    }
  }
}
```

Server responds immediately with `CreateTaskResult`:
```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "task": {
      "taskId": "786512e2-9e0d-44bd-8f29-789f320fe840",
      "status": "working",
      "statusMessage": "The operation is now in progress.",
      "createdAt": "2025-11-25T10:30:00Z",
      "lastUpdatedAt": "2025-11-25T10:40:00Z",
      "ttl": 60000,
      "pollInterval": 5000
    }
  }
}
```

### Polling Task Status (`tasks/get`)

```json
{
  "jsonrpc": "2.0",

  "params": {
    "taskId": "786512e2-9e0d-44bd-8f29-789f320fe840"
```

Blocks until task reaches terminal status. Returns the same result the original request would have returned.

```json

  "params": {
    "taskId": "786512e2-9e0d-44bd-8f29-789f320fe840"
  }
}
```

Response (for a tools/call task):
```json
{
  "jsonrpc": "2.0",
  "id": 4,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "Current weather in New York:\nTemperature: 72F\nConditions: Partly cloudy"
```

```json
{
  "jsonrpc": "2.0",
  "id": 6,
  "method": "tasks/cancel",
  "params": {
    "taskId": "786512e2-9e0d-44bd-8f29-789f320fe840"

```

### Task Data Type

```typescript
interface Task {
  taskId: string;
  status: "working" | "input_required" | "completed" | "failed" | "cancelled";
  statusMessage?: string;
  createdAt: string;    // ISO 8601
  lastUpdatedAt: string; // ISO 8601
  ttl?: number | null;   // milliseconds from creation, null = unlimited
  pollInterval?: number; // milliseconds between polls
}
```

### Related Task Metadata

All task-related messages MUST include in `_meta`:
```json
{
  "_meta": {
    "io.modelcontextprotocol/related-task": {
      "taskId": "786512e2-9e0d-44bd-8f29-789f320fe840"
    }
  }
}
```

### Tool-Level Task Negotiation

In `tools/list` response, each tool declares:
```json
{
  "execution": {
    "taskSupport": "optional"
  }
}
```

Values:
- `"forbidden"` (default) -- clients MUST NOT use task augmentation
- `"optional"` -- clients MAY use task augmentation
- `"required"` -- clients MUST use task augmentation; server returns `-32601` otherwise

This is a fine-grained layer ON TOP of `capabilities.tasks.requests.tools.call`. If the capability is absent, `taskSupport` is ignored.

## 10. Error Response Format

### JSON-RPC Error Response Structure

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32602,
    "message": "Unsupported protocol version",
    "data": {
      "supported": ["2024-11-05"],
      "requested": "1.0.0"
    }
  }
}
```

Fields:
- `code` (number, required): integer error code
- `message` (string, required): human-readable description
- `data` (any, optional): additional error context

### Standard JSON-RPC Error Codes

From schema.ts:

| Constant          | Code    | Meaning                   |
|-------------------|---------|---------------------------|
| `PARSE_ERROR`     | -32700  | Invalid JSON              |
| `INVALID_REQUEST` | -32600  | Not a valid JSON-RPC request |
| `METHOD_NOT_FOUND`| -32601  | Method does not exist      |
| `INVALID_PARAMS`  | -32602  | Invalid method parameters  |
| `INTERNAL_ERROR`  | -32603  | Internal server error      |

### MCP Implementation-Specific Error Codes (range: -32000 to -32099)

| Constant                     | Code    | Meaning                                |
|------------------------------|---------|----------------------------------------|
| `URL_ELICITATION_REQUIRED`   | -32042  | Server requires client to provide info via URL elicitation |

### MCP-Specific Error Code Usage

| Context                | Code   | Usage                                        |
|------------------------|--------|----------------------------------------------|
| Unknown tool           | -32602 | `tools/call` with nonexistent tool name      |
| Resource not found     | -32002 | `resources/read` for nonexistent URI         |
| Invalid params         | -32602 | Missing required arguments, malformed request|
| Internal error         | -32603 | Server-side failure                          |
| Invalid cursor         | -32602 | Invalid pagination cursor                    |
| Task not found         | -32602 | `tasks/get` with nonexistent taskId          |
| Task already terminal  | -32602 | `tasks/cancel` on completed/failed/cancelled |
| Task augmentation required | -32600 | Non-task request when receiver requires it |
| Unsupported protocol version | -32602 | During initialization               |

### Tool Execution Errors (not protocol errors)

Returned inside a successful JSON-RPC response with `isError: true`:

## Appendix: JSON-RPC Base Message Types

Request (MUST have id, MUST NOT be null):
```json
{ "jsonrpc": "2.0", "id": 1, "method": "string", "params": {} }
```

Notification (MUST NOT have id):
```json
{ "jsonrpc": "2.0", "method": "string", "params": {} }
```

Result response:
```json
{ "jsonrpc": "2.0", "id": 1, "result": {} }
```

Error response:
```json
{ "jsonrpc": "2.0", "id": 1, "error": { "code": -32600, "message": "string", "data": null } }
```

- IDs MUST be string or integer, MUST NOT be null.
- IDs MUST NOT be reused within the same session.

## Appendix: `_meta` Key Naming Convention

Prefix format: reverse DNS + slash (e.g., `com.example/key-name`).
Reserved prefixes: any where second label is `modelcontextprotocol` or `mcp` (e.g., `io.modelcontextprotocol/`, `dev.mcp/`).
