# Changelog

## Unreleased

### Collector compatibility and lifecycle safety

- **Strict ingest-v3 collection origin.** Every artifact now carries canonical `host_id` and `network_realm_id` provenance. Artifact-emitting collector operations require stable origin configuration, and one server database pair admits one exact host/private-network vantage point before generic validation or lifecycle writes. This prevents local paths, loopback/private endpoints, host facts, and coverage roots from being silently merged or retired across unrelated collectors. A mismatch returns `409 COLLECTION_REALM_MISMATCH`; origin IDs are explicitly non-secret provenance, not artifact authentication.
- **Immutable PostgreSQL/Neo4j storage pairing.** A canonical `storage_pair_id` and the admitted origin are stamped into both product-empty stores under a versioned binding marker. Startup serializes initialization with a PostgreSQL advisory lock and inspects both markers before migrations or graph-schema mutation; crossed pairs, future marker versions, mismatches, and unbound non-empty stores fail closed. One-sided crash recovery is allowed only while both stores remain product-empty. Every ingest rereads both markers before any audit/lifecycle/graph write and returns a sanitized `503 STORAGE_BINDING_UNAVAILABLE` on verification failure.
- **Fail-closed project discovery.** `--project-dir` is now the single validated root for project config, nested Cursor rules, and MCP auto-discovery. Config files are read once through shared bounded parsers, legitimate multi-client settings retain every applicable view, ambiguous exports do not invent a client identity, and read/parse/traversal failures cannot publish authoritative absence.
- **Evidence-backed instruction ownership.** Every project-local `.cursor/rules` tree is discovered within bounded, symlink-safe traversal that prunes `.git`. Rule nodes are owned by their stable tree domain, per-file outcomes remain diagnostic, and speculative `LOADS_INSTRUCTIONS` edges are no longer emitted.
- **Port-neutral network fingerprinting.** Every registered fingerprinter runs once per open endpoint through a bounded HTTP pool; port mappings are ordering hints only. Transport, TLS, timeout, redirect, authentication, transient-response, body-read, and matcher failures propagate as partial coverage so a transient probe cannot retire a prior service. Implementation-specific Ollama, Open WebUI, LangServe, and Qdrant probes reject real cross-service near misses without probing oversized generic schemas.
- **Real protocol corrections.** vLLM uses its service-specific `/version` endpoint with upstream version evidence and unknown authentication posture; MCP collection uses canonical per-server scopes accepted by the production ingest validator; configured and live MCP claims remain distinct during safe multi-owner projection; and GGUF parsing requires the literal `GGUF` signature with a checksum-pinned fixture generated by the upstream Python writer.
- **Cross-collector credential identity.** Recognized HTTP authorization schemes are excluded from credential material before hashing, so `Authorization: Bearer <token>` correlates with the same token observed directly by a campaign or looter instead of hashing transport syntax into a different identity.
- **Origin-correct Scan Manager commands.** Every artifact-producing command shown by the frontend's New Scan dialog now includes the required `--host-id` and `--network-realm-id` provenance flags, explains that they must match the database pair's admitted host/private-network realm, and keeps the longer copyable commands readable in the modal.

### ContextForge-backed MCP mutation and recovery

- **Provider-specific contract, not a generic MCP update API.** Tool-description mutation now requires `--adapter contextforge`; the fabricated generic management method/path/status/row-ID flags and default `/admin/tools/{id}` workflow are removed with no compatibility aliases. The positional target must be a server-scoped ContextForge MCP URL ending in `/servers/<server-uuid>/mcp`. AgentHound accepts ContextForge v1.0.5 server/tool IDs only in their lowercase 32-hex wire form and derives the deployment root while preserving an optional reverse-proxy prefix; `--management-url` can override only that root, and AgentHound appends fixed `/v1` routes.
- **Protocol-correct observation and fixed management surface.** MCP descriptions are observed through an initialized, paginated official SDK session. The adapter binds the exact MCP tool name to one server-associated ContextForge row using fixed `GET /v1/servers/<server-uuid>/tools` and `GET /v1/tools/<tool-uuid>` reads, then performs at most one `PUT /v1/tools/<tool-uuid>` with `{"description":"..."}`. It expects exact `200` JSON, rejects redirects and malformed/oversized/non-JSON responses, and never invents a standard or typical MCP admin endpoint.
- **Independent authentication and authorization.** `AGENTHOUND_MCP_TOKEN` explicitly supplies the MCP bearer token; when unset, every local MCP client-config `Authorization` header for the exact positional URL is compared and conflicting values fail closed. `AGENTHOUND_CONTEXTFORGE_TOKEN` independently overrides management authentication; without it, same-origin management reuses the resolved MCP bearer and cross-origin management fails closed. ContextForge v1.0.5 permission claims are only a token ceiling, so AgentHound accepts a session token or an API token with an empty/wildcard ceiling, then resolves the provider profile. Non-admins must prove effective `servers.read`, `tools.read`, and `tools.update` RBAC through team-context permission reads and be the direct `ownerEmail` of both server and tool; team membership alone is insufficient. A provider-authenticated platform admin uses ContextForge's admin bypass without fabricated role claims. Exact non-wildcard API-token ceilings fail closed because they block Layer-2 proof. Credentials are origin-bound and never persisted or logged.
- **Typed, single-write recovery.** The new receipt records its type/version/provider contract, separate URLs and identities, fixed routes/method/status, original and updated descriptions, original ContextForge version, and unique forward/restore operation User-Agents, but no credentials. Unknown, missing, or old receipt types/versions/profiles are rejected without networking. The receipt is durably persisted before the one forward PUT; bounded read-only polling then requires both ContextForge and MCP to show the updated value and never repeats the write. Forward and restoration bodies, exact tool records, and projected responses are bounded before mutation.
- **Ambiguity, normalization, and restoration.** A lost write response is accepted as forward success only when read-only reconciliation finds the exact intended description, `V+1`, and the pre-recorded forward User-Agent. Provider normalization or failed MCP projection is a failed forward result with one inline cleanup, not false success. Recovery may restore normalized landed text when the exact tool UUID, `V+1`, and forward User-Agent prove attribution; outbound-text equality is not required. An original-looking `V+1` forward-attributed landing completes the restore transition instead of falsely no-oping. Null/non-string originals and live AgentHound forward-operation stacks fail closed before another write. Association drift does not block an exact-row restore but makes MCP verification unavailable. ContextForge and intervening proxies must preserve operation User-Agents.
- **Provider validator limitation.** ContextForge v1.0.5 exposes no non-mutating update-validation endpoint. A stored description created under an older or permissive validation policy can remain readable while the current update path rejects that same text, so historical validator-configuration drift can make exact restoration fail; AgentHound does not disguise a mutating no-op PUT as preflight. Compile-time `Reverter` conformance guarantees an implemented recovery path, not provider acceptance at runtime. This limitation is documented separately from the residual readâ†’write race.
- **Recovery diagnostics preserve TLS intent.** When `--insecure` was explicitly used for poisoning, both successful and indeterminate CLI recovery hints now include `revert ... --insecure`; the flag remains absent from receipts and credentials remain unpersisted.
- **Truthful eligibility and restoration reporting.** Dry-run now applies the same read-only live-row and receipt eligibility guards as commit, so it cannot plan a mutation that commit would immediately reject. Association-read errors require full dual-surface cleanup verification instead of being treated as detachment. Proven association drift still permits exact-row restoration, but standalone recovery reports `PARTIALLY VERIFIED` because MCP verification is unavailable. Current SDK, CLI, sentinel, README, and module-authoring text describe an implemented recovery path with a verified runtime outcome rather than unconditional restoration; `extract` is correctly documented as non-mutating and outside Reverter.
- **Explicit concurrency limit.** ContextForge exposes no conditional update used by this adapter, so version/User-Agent checks provide post-hoc ownership evidence rather than compare-and-swap. Both forward mutation and restoration retain a final-readâ†’unconditional-PUT TOCTOU window; operators must use an exclusive operation window until the provider offers server-side preconditions. TLS remains strict by default, with `--insecure` applying explicitly to both provider surfaces and recovery.
- **Cleaner round-trip UX.** `mcp-poison-roundtrip` now takes the MCP URL, `--adapter contextforge`, the MCP tool name, and an optional management-root override. AgentHound generates the benign run-specific mutation marker internally, proves the intermediate change through both surfaces, and then proves restoration; no operator mutation text is required.

### Predicted-to-verified campaign runner

Adds an ordered, reversible campaign runner that turns *predicted* graph relationships into *observed* evidence and validates the reversible-mutation machinery end to end. The collector cannot link Neo4j, so the server exports a stable, sanitized **witness** that the collector-side `agenthound campaign` runner verifies and the server re-correlates on ingest.

**Safety / rollback foundation**

- ContextForge-backed `mcp.poison` `Revert` is conflict-**aware** and provider-specific: it restores only an exact tool-UUID/version/operation-User-Agent attribution, including provider-normalized landed text, and reports MCP verification unavailable rather than blocking restoration when association drifted. The provider lacks CAS, so residual final-readâ†’write TOCTOU exists in both forward and restore directions.
- `agenthound revert` walks each module's receipts **newest-first (per-file LIFO)** on a bounded, non-cancellable context, retains receipts on failure, and reports `INCOMPLETE` / exits non-zero unless every receipt reverted.

**Witness export (server)**

- `agenthound-server witness --finding <id>` (and `GET /api/v1/analysis/findings/{id}/witness`) export a stable, content-addressed witness â€” credential/server/resource node IDs, `value_hash` + `merge_key`, predicted edge kind, path topology, and the publication/schema revision. No Neo4j relationship IDs, no arbitrary node properties, no secrets.

**Scenarios (`agenthound campaign`)**

- **`cred-reach`** â€” a READ-ONLY differential credential-reach oracle: an unauth control probe vs. a hash-matched authed probe of the exact predicted resource, classified across the full differential matrix. Credential material is supplied out of band (env/stdin), hash-matched locally, and never logged or written to the graph. A `credential_gated_reach_verified` outcome emits `CREDENTIAL_REACH_VERIFIED`, which the server re-correlates to **upgrade the existing `CAN_REACH` finding** (no duplicate finding, no risk double-count); anonymous access emits the `PUBLIC_ACCESS_OBSERVED` fact.
- **`mcp-poison-roundtrip`** â€” a STANDALONE ContextForge target-mutation validation: an operator-authorized, internally generated benign mutation â†’ dual-surface injected-state re-read (ORACLE) â†’ conflict-aware revert â†’ dual-surface restored-state re-read (CLEANUP), with the oracle and cleanup outcomes reported **separately** and computed independently. It is **not** an attack finding and makes **no** claim about a predicted credential path; it emits **no** graph edge (the round-trip evidence stays in the campaign transport) and reuses the per-file-LIFO + conflict-aware rollback.

**Schema**

- New raw edge kinds `CREDENTIAL_REACH_VERIFIED` (AgentInstanceâ†’MCPResource) and `PUBLIC_ACCESS_OBSERVED` (MCPServerâ†’MCPResource); collector-produced edge kinds go 18 â†’ 20 (32 total with composites). The reversible round-trip deliberately adds no edge kind.

**Guardrails**

- New collector-safe packages `sdk/campaign`, `modules/credreach`, and `modules/mcproundtrip` are added to the collector allowlist; the lean binary still links no `chi`/`pgx`/`neo4j`/`server/internal` code. `--commit` is off by default (dry-run plans only) and an `AUTHORIZED` acknowledgement gates the first run.

### Evidence-backed posture: complete-revision publication + preserved partial states (#78)

Publishes only complete posture revisions and preserves unknown, partial, and error states across collection, analysis, the REST API, and the UI. Removes the pre-launch compatibility paths that let an incomplete scan masquerade as a full one.

**Collection contracts (new `sdk/ingest` primitives)**

- **`OutcomeState` vocabulary** (`sdk/ingest/metadata.go`): every collection outcome is one of `unknown`, `not_applicable`, `complete`, `partial`, `failed`, or `truncated` â€” distinguishing a *successfully empty* observation from an unattempted, failed, partial, or truncated one. `CollectionReport` / `CollectionOutcome` / `AggregateOutcomeState` roll per-target outcomes up to a single scan-level state; a `not_applicable` optional method never drags the enclosing collection out of `complete`.
- **Coverage ownership keys** (`sdk/ingest/observation.go`): `CanonicalCoverageKey(collector, scopeKind, scope)` hashes the scope identity (`collector:scopeKind:sha256:â€¦`) so target URLs, command lines, and local config paths stay out of Neo4j ownership tokens and exported metadata. `CollectorRootCoverageKey` / `ParentCollectorRootKey` give exhaustive and targeted runs a stable shared root, and `CoverageRoot` lets an exhaustive run declare the complete active child set it observed (targeted runs must not).
- **Normalization + stage results** (`sdk/ingest/result.go`): `NormalizationStatus` and per-`StageResult` reporting make ingest's own processing completeness a first-class, inspectable fact.

**Ruleset provenance**

- **`RulesetManifest`** (`sdk/rules/manifest.go`): each scan records the exact effective text- and fingerprint-rule semantics as content digests, with `Authenticity: "unverified"` unless a separate trusted-signature workflow attests the artifact. Surfaced in the UI (`features/rules/model/provenance.ts`).

**Ingest, publication, and analysis**

- **Strict-v2 ingest**: canonical collector-root coverage keys are accepted from collector scans, and a later complete invocation can repair matching *dirty* coverage left by an earlier partial run rather than leaving the graph half-updated.
- **Publication store** (`server/internal/appdb/publication_store.go`): scan finalization (`FinalizeScanParams` â†’ `PublicationResult`) publishes a new posture *revision* only when the run is complete, records observation completeness and resolved / dirty coverage, and produces a revision-scoped posture export. Findings, analysis paths, exports, and risk calculations are now current-snapshot and completeness-aware.
- **Evidence graph** (`server/internal/analysis/evidence_graph.go`): attack-path evidence is finalized into a self-consistent node/edge graph, flagging edges whose endpoints are missing from the path so a finding can't cite a broken chain.

**Schema (endpoint widening â€” no new node/edge kinds)**

- `CAN_REACH` target set widened `{MCPResource}` â†’ `{MCPResource, Credential}`; `EXPOSES` widened to source `{AIService, OpenWebUIInstance}` and target `{AIService, OllamaInstance}`.

- New label helpers: `PublicNodeLabels` (inventory ordering â€” concrete kinds before the `AIService` umbrella), `UmbrellaCompanions` (the only valid multi-label tuples), and `ConcreteNodeKind` (the sole identity-owning kind in a tuple). ## v0.7.0

Network `scan` / `discover` usability and hardening. Adds progress and summary output to the network verbs, fixes a `:MCPServer` duplicate-node bug at the cross-collector merge point, and adds memory-safety guards (absolute host ceiling, nested-include rejection, concurrency clamp) across the scan/discover path.

### Scan & discover output (#62)

- **One-line summary by default.** `scan` prints `[scan] <spec>: N host(s) with at least one open port` and `discover` prints `[discover] <spec>: N endpoint(s)`, followed (for `scan`) by per-match fingerprint lines and a fingerprint summary. The full per-host / per-endpoint listing â€” which can run to thousands of lines on a large sweep â€” is gated behind `--verbose`.
- **Interactive progress line.** When stderr is a TTY, a single rewriting line tracks the port sweep and the fingerprint phase. It is omitted automatically when output is piped or redirected, so CI logs and piped output stay clean.
- **`--quiet` / `AGENTHOUND_QUIET=1`** suppresses the progress line, the summary, and the fingerprint output, leaving only errors. Progress and summaries always go to stderr and never affect the JSON written to `--output`.

### Network scanner & discovery hardening (#64)

Fixes from a focused review of the network-level `scan` / `discover` path:

- **`discover` MCP nodes now merge with the `mcp` / `config` collectors.** Protocol discovery preserves the matched JSON-RPC path (`/` vs `/mcp`) and derives the `:MCPServer` node ID via `ingest.ComputeMCPServerID` (the full, trailing-slash-normalized URL) instead of hashing a path-less base URL. Previously a server found by `discover` got a different deterministic ID than the same server from a config/MCP scan, producing duplicate `:MCPServer` nodes at the documented merge point. A root (`/`) match canonicalizes to the path-less `host:port` form; `/mcp` is preserved.
- **Absolute host ceiling on CIDR expansion.** A hard cap of 1,048,576 hosts (exactly IPv4 `/12`, IPv6 `/108`) now applies *even with* `--allow-large-cidr`, so an override can no longer request an unbounded enumeration â€” a standard IPv6 `/64` or `0.0.0.0/0` previously walked toward OOM. Oversized specs are refused before any allocation.
- **Network-mode probe timeout honors the intended 3s default.** The shared `--timeout` flag (120s, tuned for the legacy per-server MCP/A2A collectors) no longer silently overrides the per-TCP-connect-probe timeout; network mode falls back to `networkscan`'s 3s default unless `--timeout` is set explicitly.
- **Targets files reject nested includes.** A `@file` / `file://` line *inside* a targets file is refused, preventing a self-referential or cyclic file from recursing through expansion until it exhausts memory / file descriptors.
- Corrected a misleading comment on the `AUTHORIZED` prompt (it always prompts when `--allow-public-targets` is set and is fail-closed; it never skipped private specs as the comment claimed).

Robustness and cleanup from the same review:

- **Ctrl-C now flushes partial results for `scan` / `discover`.** Both verbs wire SIGINT/SIGTERM into the scan context (`signal.NotifyContext`), so an interrupted sweep cancels the worker pool and writes the partial envelope to `--output` instead of dying mid-write. `scan` skips the fingerprint phase on interrupt; `discover` emits the endpoints found so far.
- **Overlapping/duplicate targets-file entries are de-duplicated** (order-preserving), eliminating duplicate `Target`s and redundant probes/fingerprint requests.
- **Worker-pool concurrency is clamped** (`MaxConcurrency = 4096`) in both scanners, preventing an absurd `--network-scan-concurrency` from overflowing the tasks-channel size on 32-bit or allocating a runaway buffer / goroutine count.
- **MCP discovery handles Streamable HTTP**: the probe sends `Accept: application/json, text/event-stream` (some servers `406` otherwise) and parses an SSE-framed `initialize` response, improving recall without affecting precision.
- Removed dead code: the unused `protoscan` discover registrations (`mcp.discover` / `a2a.discover`, never dispatched â€” `discover` constructs its scanner directly) and an unused error/`slog` shim. The fingerprint progress denominator now applies the same `Fingerprinter` type assertion as dispatch, and each fingerprinter receives a per-probe copy of `Meta`.

- README "Why AgentHound" now surfaces full-stack coverage â€” protocols (MCP, A2A), services, and agent clients â€” rather than implying single-protocol scope. (#63)

## v0.2.0

Offensive primitives milestone â€” turns AgentHound's credential-chain pitch from theory into a recordable demo. The architecture seam this introduces (multi-label nodes, `value_hash` cross-collector merge, fingerprinter dispatch via the rules engine) is the substrate every v0.3+ collector and Looter builds on.

### Network scanner

- New `agenthound scan <CIDR|host|@file>` mode â€” TCP port sweep against a fixed AI-service port set (Ollama 11434, LiteLLM 4000, vLLM/LangServe 8000, Qdrant 6333, MLflow 5000, Jupyter 8888, Open WebUI 3000) with a fixed-size worker pool (default 50). Implemented in `modules/networkscan/`.

### Two new fingerprinters

- **Ollama fingerprinter** (`modules/ollamafp/`) â€” GET `/api/version` returning `{"version": "X.Y.Z"}`. Captures version into the emitted node.

## v0.2.0-pre (post-v0.1 trunk consolidation)

The post-v0.1 trunk reshaped AgentHound around a two-binary split and a single-user analysis posture before the v0.2 offensive primitives shipped. The most consequential changes:

### Two-binary split

- AgentHound now ships as `agenthound` (lean field collector, ~9 MiB stripped on linux/amd64) and `agenthound-server` (Neo4j-backed analysis server with embedded React UI). See `docs/adr/0001-two-binary-split.md`.
- The collector links no Neo4j driver, no Postgres driver, no chi router, and no `server/internal/` code. CI gates the boundary via `scripts/deps-check.sh` and the linux/amd64 stripped size via `scripts/size-check.sh` (baseline + 10%).
- Collector writes scan JSON to a file or stdout. Three ingest paths to the server: file copy + `agenthound-server ingest <file>`, stdin pipe (`agenthound scan --output - | ssh op-box 'agenthound-server ingest -'`), or UI drag-drop import.

### Authentication removed at the application layer

- Removed: bcrypt password hashing, JWT sessions, `ah_`-prefixed API tokens, RBAC (admin/analyst/viewer), first-run admin user, audit logging, all `users` / `api_tokens` / `audit_log` Postgres tables, and the corresponding API endpoints (`/api/v1/auth/login`, `/auth/tokens`, `/auth/users`, `/audit/*`).
- Replaced with: a single 32-byte localhost bearer token, auto-generated at first server start, persisted to `~/.agenthound/server.token` (0o600), gating only the seven mutating endpoints (`POST /ingest`, `POST /query`, `POST /scans`, `DELETE /scans/{id}`, `POST /analysis/shortest-path`, `POST /analysis/all-paths`, `POST /analysis/weighted-path`). Path is overridable via `AGENTHOUND_TOKEN_PATH` or `XDG_CONFIG_HOME`.
- Read endpoints remain open. CORS uses `AllowCredentials: false` so a hostile origin cannot exfiltrate the token via a credentialed fetch.
- The 127.0.0.1 default bind is the security control. See `docs/security.md`.

### Rate limiting removed

- All rate limiting (httprate-based 100/min general, 20/min ingest, 10/min raw Cypher) was removed (commit `e97e043`). Single-user posture; the network layer is the access control.

### Frontend stack migrated to React Flow + ELK

- Graph visualization migrated from Sigma.js + graphology + ForceAtlas2 to `@xyflow/react` 12.6 + `elkjs` 0.9.3. DOM-based, suitable for the small-to-medium graphs typical in attack-path views.
- Dashboard reworked: ByteArmor-inspired dark theme; new widgets (exposure score, credential gauge, cross-protocol stat); risk distribution replaced with findings-by-category treemap; added Top Risky Entities; per-widget info tooltips.

### Detection rules engine

- Detection logic migrated from hardcoded Go functions to a YAML-based rules engine. 30 builtin rules in `sdk/rules/builtin/*.yaml` covering capability classification, credential extraction, prompt-injection patterns, instruction-file poisoning, and resource-sensitivity. Rules wired through every collector.
- New CLI: `agenthound rules list|validate|test`.
- New API: `GET /api/v1/rules`, `GET /api/v1/rules/{id}`.
- New UI page: Detection Rules viewer.

### CLI surface

- Stub verbs `agenthound loot|extract|poison|implant` added â€” print "not yet implemented" today, reserve the verb space for the v0.2 offensive-primitives milestone.

### Hardening

- CSRF lockdown via the localhost-token gate; OpenAPI spec at `server/internal/api/handlers/openapi.yaml` reconciled against the actual route map (CI-checked via `diff`).
- Race fix in the collector pipeline; UUID scan IDs.
- EDR-bait test fixtures moved out of the runtime `//go:embed builtin` path to `sdk/rules/builtin_tests/` so attacker-shaped strings never ship in production binaries.
- Loopback-bind enforced on the standard Docker image.

### Infrastructure

- GoReleaser v2 with cosign keyless signing and syft SBOMs. Two builds, two Homebrew formulas (`agenthound`, `agenthound-server`), multi-arch Docker images.

## v0.1.0 (2026-04-08)

Historical record of the original single-binary AgentHound release. **Several capabilities listed below have since been removed or reshaped in the post-v0.1 trunk** â€” see the Unreleased section above for what ships today. Specifically: the multi-user authentication system, RBAC roles, audit logging, and rate limiting were removed; the Sigma.js frontend was replaced with React Flow + ELK.

Initial release. AgentHound is a BloodHound-style security tool for AI agent infrastructure.

### Collectors

- **Config Collector** â€” parses 12 MCP client configuration formats (Claude Desktop, Claude Code, Cursor, VS Code, Windsurf, Continue, Zed, Cline, JetBrains, Kiro, Amazon Q, Augment) to discover agent-server trust relationships, credentials, instruction files, and host information

- **A2A Collector** â€” fetches A2A Agent Cards (v0.3.0 and v1.0) via HTTP, verifies JWS signatures (RFC 7515), scores auth posture, supports domain discovery (`--discover-domain`)

- Neo4j 4.4+ with auto-detected schema syntax (4.4 `ON...ASSERT` / 5.x `FOR...REQUIRE`)

- Deterministic content-based SHA-256 node IDs for cross-collector merge
- Batch MERGE writes with UNWIND (1000 ops/txn)
- APOC Dijkstra with non-APOC fallback for weighted pathfinding

### Ingest pipeline

- JSON schema validation, canonical snake_case properties, deduplication by objectid
- Rug pull detection via description_hash tracking across scans
- Stale composite edge cleanup scoped to source collector
